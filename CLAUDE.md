# ai-product-check-demo-lh 项目约定

## 项目定位

产品库日常需求交互原型仓库，GitHub Pages 静态部署。每个需求包含：Demo原型 + 飞书PRD文档。

## 开发规范

- 所有 Demo 为纯静态 HTML（数据硬编码，无需后端）
- 新需求流程：创建 `{功能}-demo.html` → 更新 `index.html` 导航卡片 → push main
- 命名规范：`{功能}-demo.html`，小写中划线
- 仅使用 `main` 分支，直接 push 部署
- PRD 输出到飞书知识库对应节点

## PRD 输出规范

所有需求 PRD 文档遵循以下结构，输出为飞书文档格式：

1. **一、需求背景** — 痛点描述（说清楚谁遇到了什么问题） + 原始需求/背景文档链接引用
2. **二、需求目标** — 3-5 条 bullet 说明本期要做什么
3. **三、功能细节和原型 Demo**
   - Demo 地址（GitHub Pages 在线链接，H3 标题）
   - 按功能模块拆分 H2（如 3.1 检索工作台、3.2 推送清关）
   - 每个模块下用 H3 细分（输入区域、数据来源与接口、结果展示等）
   - 字段/交互用 bullet 逐条列清楚
   - 权限说明单独 H3 列出，明确到角色
4. **四、2 期规划** — 后续功能展望
5. **五、后端确认项** — 需要后端评估/确认的接口、表结构、并发限制等
6. **六、排期建议** — 前后端拆分工作项（不填具体天数，由开发评估）

### 附加要求

- Demo 必须先于 PRD 完成，PRD 中引用 Demo 在线地址
- 日志/记录格式需给出具体示例（含字段和格式）
- 权限控制必须明确到角色（如：关务专员、产品开发）
- 接口说明需包含：调用方式（同步/异步）、传参、返回、耗时预估
- 有原始需求背景文档的，用飞书文档引用关联
